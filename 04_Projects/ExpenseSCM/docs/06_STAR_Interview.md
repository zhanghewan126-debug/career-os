# STAR Interview｜ExpenseSCM

> 按 STAR（Situation / Task / Action / Result）结构整理的面试故事，用于回答"讲一个你遇到的挑战"类问题。使用前请将 `【 】` 中的占位内容替换为你记忆中的真实数据，没有把握的数字可以改用"显著减少 / 明显提升"等定性表达。

---

## Story 1：从"账上有货"到"可用量可计算"——Supply Layer 设计

**Situation（情境）**

系统上线初期，库存只有一张在库数量表。请购人员经常反馈"明明看库存是零，结果一周后发现上周已经下过 PO 只是还没到货"，导致重复请购；仓库那边则反映"账上有货，但一问才知道已经被别的部门核准领用只是还没发料"，实际可用量和账面数字对不上。

**Task（任务）**

我需要设计一套数据模型，让"可用量"不再依赖人工在多个单据之间核对，而是能被系统直接计算出来，同时不能因此拖慢请购、采购这些高频操作的响应速度。

**Action（行动）**

- 分析了物料从请购到入库的完整生命周期，识别出 PR、PPR、PO、验收在途、检验中、已入库、验退中一共七个明确的状态节点
- 设计了 `EXP_MTL_SUPPLY` 供应层表，用 `supply_type_code` 统一表达这七层状态，每个业务节点核准/过账时同步维护数量从上一层扣减、下一层增加
- 同时设计 `EXP_MTL_DEMAND` 需求层表，记录已核准但尚未发料的开放需求，避免"账上有货但已被预约"的情况被忽略
- 权衡过用实时聚合视图的方案，但考虑到请购页面是高频查询场景，改为在过账时间点同步维护实体表，把计算成本放在低频的过账路径而不是高频的查询路径
- 推动请购和领用校验逻辑统一改为基于"可用量 = ONHAND + 在途供应 − 开放需求"这个公式判断，而不是各自读取不同的数字

**Result（结果）**

- 请购人员可以直接在系统里看到某个料号"在库 + 在途"的合并数量，【重复请购的情况明显减少】
- 仓库和采购在催料时可以直接定位物料卡在供应链的哪个具体阶段（PPR 阶段还是已到货待检验），不再需要跨系统人工核对
- 这套模型后来也成为库存事务三表对账（事务流水、在库明细、供应层）设计的基础，详见 [数据库设计文档](04_Database_Design.md#5-supply-layer-数据模型)

## Story 2：解决高并发领用场景下的库存超扣问题

**Situation（情境）**

某几种通用耗材（如办公用品、劳保用品）会被多个部门同时申请领用。早期版本中，库存扣减逻辑先在应用层查询剩余数量、校验是否充足，再执行扣减 SQL，这两步之间存在时间窗口。当同一批次被多个部门几乎同时申请时，出现过库存被扣成负数的情况。

**Task（任务）**

我需要在不牺牲太多并发处理能力的前提下，从根本上杜绝这种竞态条件导致的负库存，并且要保证问题排查起来足够直观（不能是那种偶发、难复现的 bug）。

**Action（行动）**

- 复盘发现问题根源是"查询-判断-扣减"这三步分散在应用层多次数据库交互之间，中间存在并发窗口
- 把这段逻辑收敛进一个 Oracle PL/SQL Package 存储过程，在同一个数据库事务里完成
- 对目标在库批次行使用 `SELECT ... FOR UPDATE` 加行锁，确保同一批次的并发请求会被数据库天然排队，而不是都读到同一份"未扣减前"的数量
- 额外加了一层幂等保护：以 `source_type + source_id + transaction_type_id` 建唯一约束，防止网络重试或用户手滑重复点击提交造成重复扣减
- 对比过乐观锁（版本号）方案，但考虑到这类耗材申请的冲突概率不算低、乐观锁重试会带来用户侧的报错体验，最终选择了行锁这种更符合库存扣减场景的悲观并发控制

**Result（结果）**

- 上线后【未再出现负库存问题】，相关的库存差异排查工单数量明显下降
- 因为逻辑收敛在一个存储过程里，后续同类问题（如退料入库的并发场景）复用了同一套加锁思路，减少了重复踩坑
- 这次经历也让我在后续设计验收入库、验退等同样涉及库存变动的操作时，默认就采用"数据库事务 + 行锁 + 幂等键"的组合，而不是等出问题再补

## Story 3：设计费用性物料的月度加权平均成本模型

**Situation（情境）**

财务部门反馈费用性物料的成本核算长期依赖人工在 Excel 里按月汇总入库单和验退单再手工计算，效率低、容易出错，而且跟系统里的库存数据经常对不上，月结时要花大量时间对账。

**Task（任务）**

我需要在系统里落地一套自动化的成本核算模型，既要让财务能直接从系统拿到准确的月度成本数据，又要符合费用性物料"重期间归集、不追求单笔精确"的核算特点（不同于生产物料的移动加权平均）。

**Action（行动）**

- 和财务部门确认了核算口径：按自然月为期间，期初金额+本期入库金额-本期验退金额，除以对应数量得到当期加权平均单位成本
- 设计 `ITEM_PERIOD_COST` 表，把期初、本期入库、本期验退、本期领用的数量和金额分开记录，而不是只存一个"当前成本"字段，保证任何一个历史期间都能独立复算和审计
- 明确了领用出库不参与单位成本重算，只按当期已经算好的 `unit_cost` 计价转出——这一点是和财务反复确认过的关键设计取舍，避免月内成本因为频繁领用而波动
- 实现月结流程：校验本期库存事务全部过账完成 → 按期间汇总入库/验退/领用 → 计算当期单位成本 → 回写本期出库事务的成本 → 期末数量金额结转为下期期初 → 关闭期间防止事后再过账

**Result（结果）**

- 财务月结从人工 Excel 核算改为系统直接出具，【月结耗时明显缩短】，且不再需要跟库存系统人工对账
- 期间关闭机制保证了已出具的财务数据不会被后续操作追溯篡改，满足了财务对数据稳定性的要求
- 这次和财务部门反复确认核算口径的过程，也让我更清楚"技术方案要先对齐业务规则，而不是先写代码"这件事的重要性

## Story 4：推动 Workflow 审批流与业务代码解耦

**Situation（情境）**

项目早期，不同物料类别、不同金额区间的审批路由规则是直接写在各业务模块代码里的判断逻辑。后来业务侧频繁提出审批规则调整需求（比如某类物料新增一级审批、某个金额门槛调整），每次都要改代码、测试、发版，交付周期跟不上业务节奏。

**Task（任务）**

我需要把审批规则从业务代码里剥离出来，让业务方能够在不改代码的前提下调整审批路由，同时保证业务单据的状态不会被绕过审批直接修改。

**Action（行动）**

- 分析了现有各模块里判断审批路由的代码，抽象出共性：本质上都是"根据组织、物料类别、金额，决定走哪几级审批"
- 设计业务单据统一持有一个 `audit_order_id`，关联到企业内部 Workflow Engine 的审批流表，审批节点和路由规则交给 Workflow Engine 配置管理
- 业务代码只需要在写操作前校验单据当前的审批状态（审批中 / 已核准 / 驳回），不再关心具体的审批路由细节
- 和审批流负责团队对接了状态回调机制，确保审批结果能及时同步回业务单据状态

**Result（结果）**

- 后续审批规则调整变成了配置层面的改动，不再需要业务代码发版，【交付周期从按天缩短到按小时】
- 业务单据的状态机变得更加健壮，不存在"页面直接改状态绕过审批"的漏洞
- 这套解耦思路后来也应用到了请购、采购、验收、领用、验退等所有涉及审批的模块，形成了统一的审批接入方式

## 使用建议

- 面试时优先讲 **Story 1（Supply Layer）** 和 **Story 2（并发一致性）**，这两个最能体现系统设计能力和数据库工程能力
- 如果面试官偏业务/财务背景，优先讲 **Story 3（成本核算）**，突出跨部门沟通与业务理解能力
- 如果被问到"团队协作"或"推动变革"，讲 **Story 4（审批解耦）**
- 每个故事讲完后，可以补一句"这个设计后来还应用到了哪些场景"，体现举一反三的能力
