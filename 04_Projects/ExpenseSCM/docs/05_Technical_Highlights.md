# Technical Highlights｜ExpenseSCM

> 面向技术面试的核心技术难点与解决方案整理。侧重"为什么这样设计"而非功能罗列。

---

## 1. 技术栈全景

| 分类 | 技术 | 承担职责 |
|------|------|----------|
| 后端 | Python / Flask | 业务接口、页面路由、审批流对接 |
| 前端 | EasyUI / Jinja2 / Vue.js | 表单交互、列表查询、部分模块组件化改造 |
| 数据库 | Oracle / PL/SQL | 核心业务数据、复杂事务、存储过程 |
| 数据处理 | Pandas / Excel | 报表导出、批量数据处理 |
| 系统集成 | Oracle EBS（DB Link） | 物料主档、供应商、汇率等基础数据同步 |
| 文件存储 | MongoDB | 验收附件、化学品 SGS/MSDS 等非结构化文件 |
| 工作流 | 企业内部 Workflow Engine | 多级审批、状态机驱动 |

技术选型不是追新，而是贴合企业 ERP 场景：Oracle 承担强一致性事务，Python/Flask 承担业务编排，Workflow Engine 承担审批规则的可配置化，三者分工清晰。

## 2. 核心技术难点与解决方案

### 2.1 库存事务的并发一致性

**难点**：多个使用部门可能同时申请领用同一批物料，若只在应用层判断库存是否充足，会出现"读到库存充足、扣减时却已被别人扣光"的竞态条件，导致负库存。

**解决方案**：

- 领用扣减逻辑下沉到 Oracle PL/SQL Package，在同一数据库事务内完成"查询在库明细 → 加行锁 → 校验数量 → 扣减 → 写事务流水"全过程
- 对目标批次行使用 `SELECT ... FOR UPDATE` 加行锁，锁定期间其他会话对同一批次的扣减请求会被阻塞排队，而不是并发读到同一份"过期"数量
- 按 `source_type + source_id + transaction_type_id` 建立幂等唯一约束，网络重试或用户重复提交不会产生重复扣减

**为什么不用乐观锁（版本号）**：库存扣减是高频、短事务操作，乐观锁在冲突率较高时会导致大量重试和用户侧报错体验差；悲观行锁虽然牺牲了一定并发度，但在费用性物料这种批次数不多、单批次并发申请量有限的场景下，行锁的等待时间可接受，且实现和排查都更直观。

### 2.2 多表联动事务的原子性

**难点**：验收入库、领用、验退、月结这些操作往往涉及 5 张以上业务表联动（单据表、库存事务表、在库明细表、供应层表、成本表）。如果在 Python 应用层用多次 SQL 提交拼接完成，任何一步失败都可能留下"半成功"的脏数据，且回滚逻辑分散难以维护。

**解决方案**：

- 将这类多表联动操作封装为 Oracle PL/SQL Package 中的存储过程，保证在单一数据库事务内完成
- 过程内部按步骤设置 `SAVEPOINT`，某一步业务校验失败时可以精确回滚到该步骤之前，而不是整体回滚导致不必要的重试成本
- 每个步骤失败都返回明确的错误码和描述，供 Python 层直接展示给用户，而不是抛出裸的数据库异常

**收益**：把"保证多表一致"这件事从应用层的编排责任，转移成数据库事务本身的责任，减少了 Python 侧需要处理的异常分支，也让核心业务规则集中在一处，便于后续审计和修改。

### 2.3 Supply Layer：让"可用量"可计算

**难点**：传统做法只维护一张在库数量表，只能回答"仓库里有多少"。但采购、仓库、需求部门真正关心的是"还能不能再申请"，这需要综合在库、在途、已核准未发料的需求。如果没有统一模型，这类判断只能靠人工核对多个系统或 Excel，容易出现重复采购或超发。

**解决方案**：设计 `EXP_MTL_SUPPLY` 供应层表，用 `supply_type_code` 表达物料在 PR / PPR / PO / RECEIVING / ACCEPT / ONHAND / RETURN 七个阶段的数量分布；配合 `EXP_MTL_DEMAND` 需求层记录已核准未发料的开放需求。可用量计算公式：

```text
可用量 = ONHAND + 在途供应（PR+PPR+PO+RECEIVING+ACCEPT） − 开放需求
```

业务每推进一个节点，数量在供应层内部流转（上一层扣减、下一层增加），整条链路数量守恒，详见 [数据库设计文档](04_Database_Design.md#5-supply-layer-数据模型)。

**技术取舍**：曾考虑过用视图（VIEW）实时聚合各业务表计算可用量，但涉及表多、聚合逻辑复杂，高频查询（请购校验、列表页展示）性能不理想。改为在业务过账时同步维护 Supply Layer 实体表，用"写时计算、读时直取"的方式，把计算成本从查询路径转移到过账路径，符合这类系统读多写少的访问特征。

### 2.4 月度加权平均成本核算

**难点**：费用性物料的成本核算既要满足财务月结口径的稳定性，又不能像制造业物料一样对每笔出入库都做移动加权平均重算（那样计算量大、月内成本波动频繁，不利于费用分摊对账）。

**解决方案**：采用月度加权平均法，`ITEM_PERIOD_COST` 表按物料 + 期间分别记录期初、本期入库、本期验退、本期领用的数量与金额，月结时统一计算：

```text
当期单位成本 = （期初成本×期初数量 + 本期入库金额 − 本期验退金额）
              ÷ （期初数量 + 本期入库数量 − 本期验退数量）
```

领用出库不参与单位成本重算，只按当期已计算出的 `unit_cost` 计价转出，月内成本口径稳定；期末数量金额滚动结转为下期期初，期间通过 `closed_flag` 关闭防止事后篡改。详见 [数据库设计文档](04_Database_Design.md#7-成本核算数据模型)。

### 2.5 Workflow 审批流与业务状态解耦

**难点**：企业内不同组织、不同物料类别、不同金额区间的审批规则都不一样，如果把审批路由写死在业务代码里，规则一变就要改代码发版。

**解决方案**：业务单据只维护一个 `audit_order_id` 关联审批流表，审批节点、路由规则由 Workflow Engine 独立配置管理；后端在业务写操作前统一校验单据当前审批状态，保证状态机不会被页面绕过。业务代码只关心"审批通过 / 驳回"两个结果，不关心具体走了几级、哪些人审批，审批规则调整对业务代码零侵入。

### 2.6 Oracle EBS 集成

**难点**：物料主档、供应商、汇率等基础数据存在于集团 Oracle EBS 中，ExpenseSCM 需要保证这些数据与 EBS 保持一致，同时不能让集成逻辑侵入核心业务表结构。

**解决方案**：通过 DB Link 建立与 EBS 的只读同步链路，`EXP_SYSTEM_ITEMS` 等主数据表的主键与 EBS 对应表（如 `INVENTORY_ITEM_ID`）对齐，业务表统一通过 `item_id` 引用，不需要做二次映射；同步为单向只读，避免 ExpenseSCM 的本地数据变更污染 EBS 主数据。

## 3. 性能与稳定性考虑

- **索引设计**：库存事务表、在库明细表按 `item_id + org_id` 建组合索引，配合 `source_type + source_id` 唯一索引，兼顾查询效率与幂等校验
- **批量处理**：报表导出、期末结转等批量场景使用 Pandas 做内存侧汇总，减少对 Oracle 的重复往返查询
- **行锁粒度控制**：`SELECT FOR UPDATE` 仅锁定命中的具体批次行，而不是锁整张表或整个物料，保证不同物料、不同批次的并发操作互不阻塞

## 4. 技术选型说明（面试可能被追问）

**为什么核心逻辑放 PL/SQL 而不是 Python？**
涉及多表联动的事务如果放在应用层，网络往返和事务边界都更难控制；PL/SQL Package 运行在数据库内部，事务性、锁机制、回滚控制都是数据库原生能力，更适合承担"强一致性"这类职责。Python/Flask 更适合承担业务编排、页面交互、跨系统调用这类"胶水"职责。

**为什么不引入消息队列做异步解耦？**
ExpenseSCM 的核心链路（验收入库、领用扣减）要求强一致、实时可见（仓库操作后马上要能看到最新库存），异步化会引入数据最终一致的复杂度，与业务对"账实一致"的诉求冲突，因此保留同步事务模型。

---

## 5. 技术亮点总结（面试速记）

1. Oracle PL/SQL Package + SELECT FOR UPDATE + 幂等键，解决高并发库存扣减的一致性问题
2. Supply Layer 七层供需模型，把"可用量"从人工核对变成可计算的数据模型
3. 月度加权平均成本核算，兼顾财务口径稳定性与费用性物料的核算特点
4. Workflow 审批流与业务状态解耦，审批规则可配置、业务代码零侵入
5. Oracle EBS 单向只读集成，主数据一致且不侵入核心表结构
6. 全链路 PR → PPR → PO → Receiving → Inventory → Issue → Cost 数据可追溯，任意库存数字可回溯到具体单据
