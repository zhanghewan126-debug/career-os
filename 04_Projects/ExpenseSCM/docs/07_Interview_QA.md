# Interview Q&A｜ExpenseSCM

> 围绕 ExpenseSCM 项目整理的高频面试问答，覆盖技术、数据库、业务与系统设计四个维度。

---

## 1. 技术类

**Q1：为什么核心库存逻辑放在 PL/SQL 而不是 Python 里实现？**

涉及多表联动的库存操作（验收入库、领用扣减、验退）需要强事务一致性和行级锁控制。这些能力是数据库的原生能力，放在 PL/SQL Package 里可以在同一个数据库事务内完成"加锁、校验、过账"全过程，出错时用 SAVEPOINT 精确回滚。如果放在 Python 应用层，跨多次数据库交互拼接事务，事务边界和锁的持有时间都更难控制，也更容易因为网络问题导致中间状态不一致。Python/Flask 更适合承担业务编排、页面交互和跨系统调用这些职责。

**Q2：Flask 和 EasyUI/Jinja2/Vue 混用，是怎么划分的？**

早期模块以 EasyUI + Jinja2 服务端渲染为主，交互相对简单、开发效率高；后续新模块和需要更丰富交互的部分逐步引入 Vue.js 做组件化改造。两者共存是渐进式技术演进的结果，而不是同时新建的技术选型，迁移是按模块推进、优先改造交互复杂或维护成本高的部分。

**Q3：MongoDB 在系统里承担什么角色，为什么不直接存 Oracle？**

MongoDB 用于存储验收环节的附件，比如化学品的 SGS/MSDS 证明文件、验收照片等非结构化数据。这类数据体积不固定、查询模式简单（按单据 ID 取文件），不需要事务性和复杂关联查询，放在 MongoDB 里比占用 Oracle 的 BLOB 存储更轻量，也避免了大文件拖慢 Oracle 的备份和迁移。

**Q4：系统怎么和 Oracle EBS 集成的？**

通过 DB Link 建立与 EBS 的只读同步链路，物料主档、供应商、汇率等基础数据从 EBS 单向同步到 ExpenseSCM 本地表，主键（如 `item_id`）与 EBS 对应字段对齐，业务表统一引用本地同步表，不需要做二次映射。同步方向严格单向只读，避免本地数据变更污染 EBS 主数据。

**Q5：项目里怎么做并发控制的？为什么选悲观锁不选乐观锁？**

领用扣减库存时，对目标批次行用 `SELECT ... FOR UPDATE` 加行锁，保证同一批次的并发申请被数据库排队处理而不是都读到同一份过期数量。之所以选悲观锁而不是版本号乐观锁，是因为部分通用耗材的并发申请冲突概率不低，乐观锁在冲突时需要业务重试，用户体验和实现复杂度都不如行锁直接；而库存扣减本身是短事务，行锁的等待时间可控。

## 2. 数据库设计类

**Q6：为什么要单独设计 Supply Layer，而不是只维护一张库存表？**

一张库存表只能回答"现在库里有多少"，回答不了"还有多少在途"和"账上的量有多少已经被预约"。Supply Layer 把物料在 PR、PPR、PO、验收在途、检验中、已入库、验退中七个阶段的数量分别记录，配合需求层记录已核准未发料的开放需求，可用量 = 在库 + 在途 − 开放需求，这样请购、领用校验才有一个可计算、可信的数字依据，而不是靠人工核对多个单据。详见 [数据库设计文档第 5 节](04_Database_Design.md#5-supply-layer-数据模型)。

**Q7：库存事务表、在库明细表、供应层表，这三张表是怎么配合、怎么保证一致的？**

事务表记录历史流水（只增不改，回答"怎么变成这样的"），在库明细表按批次记录当前剩余数量（回答"现在还有多少、该扣哪一批"），供应层表记录当前所处的供应链阶段（回答"除了在库还有多少在途"）。一致性约束是：在库明细剩余数量合计 = 事务表净额 = 供应层 ONHAND 层数量。这三者的同步更新都在同一个 PL/SQL Package 事务内完成，任意时点都可以交叉对账，一旦对不上就能定位到具体哪一笔事务出的问题。

**Q8：为什么领用出库不参与单位成本重算？**

费用性物料采用月度加权平均法，如果每笔领用都触发成本重算（类似移动加权平均），会导致月内成本频繁波动，财务做费用分摊时口径不稳定，也增加了计算量。设计上把"改变库存价值构成"的动作（入库、验退）和"转出已有价值"的动作（领用）分开：只有前者参与单位成本的重新计算，领用只是按当期已经算好的 `unit_cost` 计价转出，月内口径保持稳定。

**Q9：怎么防止同一张单据被重复过账？**

在库存事务表上以 `source_type + source_id + transaction_type_id` 建唯一约束作为幂等键。同一张单据的同一个动作（比如同一张验收单的入库过账）无论因为网络重试还是用户重复点击提交，都只会成功过账一次，第二次会因为唯一约束冲突被数据库拒绝。

**Q10：多组织的数据隔离是怎么设计的？**

核心业务表都带 `org_id` 字段，表名以 `_ALL` 结尾（如 `EXP_PO_HEADERS_ALL`）表示这是跨组织存储、按组织访问的模式，这是参考 Oracle EBS 的多组织建模方式。查询和写操作都按 `org_id` 过滤，保证不同组织间的业务数据互不可见。

## 3. 业务与供应链理解类

**Q11：费用性物料和生产性物料在系统设计上有什么本质区别？**

费用性物料不直接构成产品成本，采购频率高、种类杂、单价相对不敏感，核算更关注期间归集而不是单笔精确追溯；生产性物料要支撑 BOM 展开、生产订单成本归集，通常需要更精细的批次级成本追踪（如移动加权平均或标准成本）。这个区别直接影响了我们选择"月度加权平均、领用不重算"这种相对轻量但期间口径稳定的核算模型，而不是照搬生产物料的做法。

**Q12：请购、采购准备（PPR）、采购订单（PO）为什么要分成三个独立单据，而不是请购通过直接变成 PO？**

请购只表达"需要什么、需要多少"，属于需求方；采购准备阶段要补全供应商、单价、币别、税率等商务条件，属于采购方的议价和归并工作；采购订单是与供应商的正式契约。如果请购直接生成 PO，采购议价、更换供应商、多张请购拼单采购这些操作都要反过来修改需求数据，破坏需求记录的真实性。分层设计让"需求"和"商务条件"解耦，各自变更互不影响。

**Q13：收料异常（比如化学品缺 SGS/MSDS）是怎么处理的？**

验收环节对特定物料类别（如化学品）做强制校验，如果缺少必要的合规文件或数量、品质不符，系统会自动建立异常单并阻断后续的入库流程，防止不合规物料流入库存。异常处理完成前，对应批次不会计入可用库存。

## 4. 系统设计与工程实践类

**Q14：如果让你重新设计这个系统，你会改变什么？**

会更早引入 Supply Layer 这类供需模型，而不是等到库存不透明的问题暴露后再补设计；另外审批流与业务解耦这件事也应该在项目一开始就做，而不是积累了多次改代码发版的痛点之后才重构。这两点都属于"早期多花一点设计成本，换取后期迭代速度"的取舍。

**Q15：这个系统的性能瓶颈可能在哪里？你是怎么应对的？**

高频查询集中在请购校验（读可用量）和列表查询（按物料、组织、状态筛选）。应对方式包括：Supply Layer 用实体表而不是实时聚合视图，把计算成本放在低频的过账路径；核心查询字段（`item_id + org_id`、`source_type + source_id`）建组合索引；批量报表和期末结转用 Pandas 在应用层做汇总，减少对 Oracle 的重复往返查询。

**Q16：怎么保证数据库设计既贴合企业需求又不过度设计？**

优先满足已明确的业务规则（比如月结口径、审批规则、多组织隔离），对不确定或变化频繁的部分（比如审批路由细节）通过配置化（Workflow Engine）而不是写死的表结构去承载，避免为了"未来可能的需求"预先设计一堆用不上的字段和表。

**Q17：你在这个项目里主要负责什么，遇到的最大挑战是什么？**

负责需求分析、数据库设计、Python 后端开发、Oracle SQL/PL-SQL 开发、Workflow 审批流开发、库存事务与成本核算实现、EBS 集成、测试上线维护全流程。最大的挑战是在保证强一致性（并发库存扣减、多表事务）的同时不过度牺牲系统响应速度和可维护性，具体做法见 [STAR 面试案例](06_STAR_Interview.md)。
